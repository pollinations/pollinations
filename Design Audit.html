Summary

The dashboard already has a real token system (dashboard-theme.ts) built around 7 page themes — amber, blue, gray, green, pink, teal, violet — but the same color shows up under four different keys (panelColors, cardColors, pillColors, tagColors, tabColors, themeTokens) and they don't always agree. The biggest wins are: collapse pill / tag / tab-active / button-light into one chip primitive, ship a single <Switch>, and promote the 4 hard-coded #hex money colors into the token file.

High

Strong-weight mapping is inconsistent across themes

"Strong" sits at different lightness per theme — amber 500, teal 600, pink 700, gray 700, indigo 900, red 900, green 950. Pick one rule (e.g. always 600 background + white text, or always 950 + 50 text) and adjust the outliers. Right now "primary on amber" reads visually lighter than "primary on green".

High

Collapse Tag / Pill / Tab-active / Button-light into one chip primitive

Per theme, four tables hold the same recipe bg-X-200 + text-X-900. They should be a single chipColors[theme] consumed by Tag, Pill, Tab (active), and Button (light). The shape (rect vs pill vs full) and behavior (clickable vs static) is what differs — not the color.

Same recipe, four definitions: tagColors.amber = bg-amber-200 text-amber-900 pillColors.amber = bg-amber-200 text-amber-900 tabColors.amber.active = bg-amber-200 text-amber-950 ← text drift buttonColors.amber.light = bg-amber-200 text-amber-900

11

Audit findings

Hand-pickable list, ordered by impact. Severity reflects how much visual debt removing it pays back.

High · 01

Three tables (Tag, Pill, Tab-active) + Button-light all encode the same chip

Per theme, four tables define bg-X-200 text-X-900. They should be one chipColors[theme] consumed by all four primitives. Estimated removal: 4 lookup tables × 7 themes = 28 redundant entries.

High · 02

dashboardThemeClasses[X].active ships text-green-950 for every theme

Active nav row stays dark-green even on the pink, blue, violet, teal, amber pages. Either drop the text class entirely or use text-X-950. Copy-paste from the green entry.

High · 03

Switch primitive duplicated

AutoTopUpToggle (balance) and DeveloperEarningsSwitch (api-keys) are the same track / thumb / animation. Promote to ui/switch.tsx with theme + status props.

High · 04

"Strong" button weight is at a different shade per theme

500 (amber), 600 (teal, violet), 700 (pink, gray), 900 (purple, red), 950 (green). Visually, the primary button doesn't read as the same UI element across pages.

High · 05

buttonColors.blue.light === buttonColors.blue.strong

Both are bg-blue-200 text-blue-900 hover:bg-blue-300. Either fix strong to use bg-blue-600 text-white for parity with teal/violet, or document that blue has no strong variant.

Medium · 06

purple and violet both exist; purple actually renders indigo

buttonColors.purple uses bg-indigo-*. cardColors.purple uses real purple. tagColors.purple uses real purple. Drop purple, keep violet as the canonical name.

Medium · 07

Tag color recipes drift from pill / theme

tag.green, tag.pink, tag.teal all use text-gray-900 while pill uses text-X-900. tag.blue is bg-blue-100 text-blue-700 — wholly different recipe. Align with the chip token from finding 01.

Medium · 08

Card and Panel are the same surface at two radii

Merge into <Surface tone="white"|"tinted" size="card"|"panel">.

Medium · 09

Money colors live as raw hex in 6 places

#E08A52, #FCD34D, #7C3F1E, #7A5807 appear as bg-[#…] / text-[#…] across pollen-balance, earnings-graph, usage-graph. Hoist to CSS vars + Tailwind tokens.

Medium · 10

Type scale leaks arbitrary px (10, 11, 15)

Used in nav (text-[15px]), brand chip / footer (text-[11px]), InfoTip badge (text-[10px]). Make them named tokens. Also: h4–h6 have no font family.

Medium · 11

InfoTip is the only place using two-color gradients

bg-gradient-to-r from-X-50 to-Y-50. Replace with flat bg-X-50 — one fewer concept, and matches the rest of the system.

Low · 12

Teal uses border-200; every other theme uses border-300

Bump cardColors.teal and panelColors.teal to border-teal-300.

Low · 13

Yellow tone in Tag and InfoTip used only there

Drop tagColors.yellow and InfoTip tone="yellow"; replace with amber.

Low · 14

Input focus ring is locked to green

The shared <Input> uses focus:border-green-500 regardless of page theme. Either accept a theme prop or remove the page-themed assumption from focus rings everywhere.

Next pass

Proposed clean shape

Once findings 01–05 are resolved, the shape of dashboard-theme.ts becomes: 

themes = {
  amber:  { hue: "amber",  ramp: { …auto-derived from Tailwind } },
  blue:   { hue: "blue",   …                                       },
  gray:   { hue: "gray",   …                                       },
  green:  { hue: "green",  …                                       },
  pink:   { hue: "pink",   …                                       },
  teal:   { hue: "teal",   …                                       },
  violet: { hue: "violet", …                                       },
}

// per-theme roles, derived (no hand-typing of "bg-X-200 text-X-900"):
text:    label | base | strong | muted | soft | softer
border:  idle  | soft | subtle
bg:      idle  | subtle | active | hover
chip:    bg + text                  // ← replaces tag/pill/tab-active/btn-light
button:  light | strong | outline   // strong rule = one shade across themes
surface: card | panel               // ← replaces Card + Panel
switch:  off | on | alert           // ← shared primitive
ring:    focus

semantic = { destructive: red, success: green, warning: amber }
money    = { paid, paid-deep, tier, tier-deep }  // 4 hex, hoisted from balance-colors.ts

With these changes the four "is this a tag or a button-light or a tab" judgement calls collapse into one (chip), and the theme story drops from 6 tables × 7–10 themes (~ 60 entries) to one derived ramp per hue.
