Visual language

Bars first

Stacked bars for composition, grouped bars for comparisons, tables for rankings and ownership.

Metric vocabulary

Quest and Paid only

No user-tier segmentation. Quest is free Pollen; Paid is purchased Pollen.

Panel contract

Every chart explains itself

Question, definition, source, caveat, freshness and unit are included in panel descriptions.

Release method

Build in parallel

New UIDs and folder. Existing dashboards remain available until explicit retirement approval.

Scope architecture

Three mutually exclusive business streams

Every successful billed generation is assigned once for revenue-bearing consumption: BYOP first, then direct community, then Core API. The Community dashboard may still show all community traffic, including its BYOP overlap, but primary revenue attribution never double-counts it.

01 · First match

BYOP Apps

api_key_client_id is present. Includes community-model calls made through BYOP.

02 · Otherwise

Direct Community

model_provider_used = 'community' and the request was not attributed to BYOP.

03 · Remainder

Core API

Authenticated generation requests that are neither BYOP nor community-model traffic.

i
Revenue language in Grafana

Paid consumption is revenue-bearing Pollen consumed. Quest consumption is subsidy usage, not revenue. Cash collected, provider cost and profitability remain in the Economics app.

Small, explicit and fast backing datasets

We will reuse proven materializations where their grain matches the metric and build only two new aggregates up front. The balance proxy benchmark proved that a shared direct query is viable for the initial 30-day dashboard, so no third balance dataset is required now.

Aggregating datasource fed from generation_event_v2.

Input filter: production plus the isolated debug-prod-copy staging fixture; dashboards must select one environment explicitly.

Grain: hour × environment × canonical model × workload × normalized funding × BYOP × community × business stream.

Measures: all calls, successful 2xx calls, billed calls, users, apps, usage units, Paid/Quest consumption and rewards.

Aggregating datasource fed from generation_event_v2.

Measures: all/successful/billed calls, users, gross consumption and developer reward.

Known shortcomings and remedies

What can mislead us—and how we contain it

The rebuild should not make a chart look more authoritative than its underlying evidence. These limitations are part of the implementation contract.

R1

Duplicate generation events

Retries can duplicate logical events. Measure financial and count impact before using additive materialized states; deduplicate or rebuild finalized periods if material.

R2

Observed balance is not total wallet stock

The proxy includes only users with billed activity that day, so it moves with wallet values and population composition. Rename it, emphasize p95/p99 concentration, and use latest D1 for exact current totals and holder tables.

R3

D1 is current-state only

The daily sync replaces the datasource. Do not imply historical account state; registration series from current D1 excludes accounts later deleted.

R4

Platform load excludes cache and anonymous traffic

v2 contains authenticated, non-cached requests. State that scope on the dashboard rather than calling it total edge traffic.

R5

Provider is configured routing, not final provider truth

model_provider_used is the registry provider and may not reflect fallback execution. Do not use it as a provider-health or exact provider-load metric.

R6

Modality cannot rely on event type alone

Image routes may represent image, video or 3D workloads. Derive presentation modality from the resolved-model registry contract.

R7

Paid consumption is not cash collected

Paid Pollen consumption is revenue-bearing usage. Stripe cash, discounts, provider cost and profit remain Economics measures.

R8

Wallet movement feeds are incomplete in Tinybird

Quest claims and authoritative purchase-credit ledgers are not yet available. Defer the wallet bridge until those sources are intentionally ingested.

R9

Tinybird staging has no D1 user snapshot

Five current-user and registration panels are query-valid but show no rows locally. Add a privacy-safe representative staging snapshot before final UI acceptance; never copy production identities or replace the production datasource for this purpose.

Definition of done

Acceptance gates

The rebuild is complete only when correctness, speed and operational clarity are demonstrated together.

Gate A

Metric integrity

Business-stream totals are mutually exclusive, Paid and Quest are never conflated, duplicate impact is controlled, and each panel has an explicit semantic contract.

Gate B

Performance

Normal 30-day queries complete under 2 seconds, supported worst cases remain safely below 10 seconds, and no dashboard fires an unbounded high-cardinality query.

Gate C

Operational usability

All five dashboards load with 30-day defaults, use consistent bar-first visuals, explain their sources and caveats, and answer distinct daily monitoring questions.
